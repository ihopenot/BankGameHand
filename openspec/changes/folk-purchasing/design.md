## Context

BankGameHand 是一个经济模拟策略游戏，使用 ECS 架构（Entity-Component-System）。公司端的采购（逐轮匹配）已实现在 `market_service.py` 和 `company_service.py` 中。居民部门由多个消费群体组成（数量和属性由配置决定），各自为独立经济实体，但采购机制与公司不同。

现有实现：
- `MarketService` 提供 `add_sell_order()`、`get_sell_orders()` 和逐轮 `match()` 方法
- `CompanyService` 通过 `sell_phase()` 挂单、`buy_phase()` 生成 BuyIntent、`settle_trades()` 结算
- `Entity` 基类 + `BaseComponent` 组件系统已完备
- `ConfigManager` 从 `config/*.yaml` 加载配置，返回 `AttrDict` 支持属性访问

## Goals / Non-Goals

**Goals:**
- 实现 Folk 实体类，配置驱动的独立经济实体（不硬编码阶层标识）
- 新增 `config/folk.yaml` 配置文件，定义 `folks: List[Folk]`，每个 Folk 条目的参数差异决定其消费行为差异
- 实现居民加权均分采购机制（基于 softmax 归一化的评分权重）
- 实现迭代重分配逻辑（卖方库存不足时重新分配）
- 实现采购结算（商品转移 + 现金支付）
- 可被主循环集成调用

**Non-Goals:**
- 收入计算（工资发放由公司端处理，本次不实现）
- 购买力约束（暂时无限现金）
- 储蓄机制和银行交互
- 经济指数更新（消费数据反馈到通胀率等）

## Decisions

1. **采购机制选择：加权均分而非逐轮匹配**
   - 居民每个群体作为一个整体消费群体，不像公司有各自独立的 CEO 偏好
   - 加权均分更符合消费者群体行为：不是全部涌向同一家店，而是自然分散
   - 评分使用 softmax 归一化，确保所有卖方（包括低评分者）都能分到一定需求

6. **配置驱动，不硬编码阶层**
   - Folk 实体不持有 tier/层级枚举，代码层面不区分"低/中/高收入"
   - 不同群体的行为差异完全由 `config/folk.yaml` 中的配置参数决定：人口、偏好权重（w_性价比、w_品牌）、各商品的人均基础需求和经济敏感度
   - 配置格式 `folks: List[Folk]`，每个条目包含 `base_demands: Dict[商品名, {per_capita, sensitivity}]`，不同群体对同一商品的人均需求可以不同
   - 这样做的好处：可以灵活调整群体数量和参数，不受"三层"的硬编码限制

2. **评分归一化：softmax**
   - `权重_i = exp(评分_i) / sum(exp(评分_j))`
   - 优势：自然处理负评分和零评分，所有卖方都参与分配
   - 高分卖方获得指数级更多的需求，但低分卖方也有份额

3. **库存不足处理：迭代重分配**
   - 某卖方库存不足以满足分配量时，只成交其剩余库存
   - 将该卖方移除候选列表，对剩余卖方重新 softmax 并分配
   - 循环直到需求满足或无卖方有库存

4. **不修改 MarketService**
   - 居民不调用 `match()` 方法
   - 直接通过 `get_sell_orders()` 获取卖方挂单，操作 `SellOrder.remaining` 属性

5. **居民暂时无限现金**
   - 简化第一阶段实现，专注采购分配机制
   - 后续可在此基础上添加购买力约束

## Risks / Trade-offs

- **softmax 温度参数**：当前未设温度参数，评分差异大时分配可能过度集中于最高分卖方。必要时后续可引入温度参数调节
- **与设计文档的偏差**：Folk.md 原文写的是"共用逐轮匹配机制"，本次实现改为加权均分，需同步更新设计文档
- **无限现金简化**：居民不受购买力约束，可能导致经济模拟中消费端过于强势，但有利于先验证采购分配逻辑的正确性
